# ACK-Free Reliable Sender Strategy for xDrip Broadcast Mode

## Scope

This document proposes a practical sender-side reliability strategy for xDrip+ **broadcast mode** using:

- **Intent action:** `com.eveningoutpost.dexdrip.NS_EMULATOR`
- **Extras:** `collection="entries"` and `data=<JSON array>`
- **Target package:** `com.eveningoutpost.dexdrip`

The design assumes:

- xDrip+ **does not send an ACK** back to the sender
- Android broadcast delivery is **not a guaranteed-delivery transport**
- Reliability must therefore be implemented **in the sender**

---

## Design Goal

Build the sender as if the transport were:

- **one-way**
- **asynchronous**
- **idempotent**
- **eventually consistent**

### Core Rule

**Do not try to make broadcast reliable by immediate duplicate spam. Make it reliable by sender-side persistence + replay window + reconciliation.**

---

## Recommended Architecture

Use a 4-layer design.

### Layer A — Persistent Outbound Store

Every glucose reading must be written to the sender's local database **before** sending the broadcast.

Store at least:

- `reading_id` — immutable unique ID generated by sender
- `timestamp_ms` — exact measurement timestamp
- `sgv_mgdl`
- `direction`
- `created_at`
- `last_sent_at`
- `send_count`
- `state`

### Layer B — Immediate Single-Send Path

When a reading is created:

1. persist it locally
2. send **one** explicit broadcast immediately

Recommended broadcast format:

```java
Intent intent = new Intent("com.eveningoutpost.dexdrip.NS_EMULATOR");
intent.setPackage("com.eveningoutpost.dexdrip");
intent.putExtra("collection", "entries");
intent.putExtra("data", jsonArrayString);
context.sendBroadcast(intent);
```

### Layer C — Periodic Replay / Backfill Worker

Run a background worker that periodically replays a **small recent time window** of readings.

This is the real reliability engine.

### Layer D — Health / Drift Monitor

Track sender-side health:

- last locally generated reading time
- last immediate send attempt time
- last replay batch time
- replay queue size
- oldest replay-eligible reading age

Raise warnings if:

- no new reading has been generated for too long
- replay worker has stopped running
- replay backlog is growing unexpectedly

---

## Delivery Contract

Treat each reading as **immutable** once created.

### Rules

1. **Never change a reading's timestamp after creation**
2. **Never synthesize a new timestamp for old data**
3. **Keep replay deterministic**

If a reading measured at `10:05` is replayed later, it must still be replayed as `10:05`, not as "now".

---

## JSON Payload Shape

Use the documented xDrip-compatible structure:

```json
[
  {
    "type": "sgv",
    "date": 1713000000000,
    "sgv": 123,
    "direction": "Flat"
  }
]
```

### Notes

- `type` must be `"sgv"`
- `date` must be milliseconds since Unix epoch
- `sgv` is glucose value in mg/dL
- `direction` must be one of xDrip's accepted trend strings

---

## Sender State Machine

Use a simple sender-side state model.

### `NEW`
Reading created locally but not yet broadcast.

### `SENT_ONCE`
Immediate broadcast was attempted.

> This does **not** mean confirmed by xDrip.

### `ELIGIBLE_FOR_REPLAY`
Reading remains available for replay until it ages out of the replay horizon.

### `ARCHIVED`
Reading is retained for audit/history but is no longer replayed.

## Replay Strategy That Actually Works

## Do Not Do This

Do **not** resend the same reading every few seconds as a panic retry.

That approach:

- does not create delivery guarantees
- increases traffic unnecessarily
- is likely to be ignored as duplicate data

## Do This Instead

Replay by **window**, not by immediate duplicate burst.

### Recommended Replay Policy

#### Immediate path
- send each reading once when created

#### Short replay cycle
- every **5 minutes**
- replay the last **2–3 readings** or all readings from the last **15 minutes**

#### Long replay cycle
- every **30–60 minutes**
- replay the last **12 readings** or all readings from the last **60 minutes**

#### Restart recovery
- on sender app restart or reboot recovery
- replay the last **60–120 minutes** once

This gives you practical eventual consistency without turning the transport noisy.

---

## Batch Size Guidance

Prefer **small batches**.

### Good sizes

- **Immediate send:** 1 reading
- **Short replay batch:** 2–6 readings
- **Cold-start recovery:** 12–24 readings

### Why

Small batches:

- reduce blast radius if a send is delayed
- match the JSON-array protocol naturally
- simplify bookkeeping and replay control

---

## Robustness Against Android Lifecycle Issues

Because broadcast delivery itself is not a durable queue, your sender must persist enough state to survive:

- app crash
- process death
- app restart
- phone reboot
- battery optimization interruptions

### Sender Requirements

- all unsent/replay-eligible readings must survive restart
- replay workers must resume automatically
- xDrip package targeting must always be explicit

Always send with:

```java
intent.setPackage("com.eveningoutpost.dexdrip");
```

---

## Practical Default Policy

If you want a simple and strong starting point, use this configuration.

### Normal operation
- persist each reading locally
- send once immediately

### Short replay worker
- every **5 minutes**
- resend all readings from the last **15 minutes**

### Long replay worker
- every **60 minutes**
- resend all readings from the last **60 minutes**

### Restart recovery
- on app restart
- resend the last **60 minutes** once

### Retention
- keep at least **7 days** of sender-side history if possible
- minimum retention: **24–48 hours**

## Minimal

- persist locally
- send once immediately
- replay last 3 readings every 15 minutes

## Better

- persist locally
- send once immediately
- replay last 15 minutes every 5 minutes
- replay last 60 minutes every hour

Everything in **Better**, plus:

## Example Sender Pseudocode

```text
onNewReading(reading):
    persist(reading)
    sendBroadcast(reading)
    markSentAttempt(reading)

shortReplayJob(every 5 min):
    readings = selectReadings(last 15 minutes)
    sendBroadcastBatch(readings)
    updateReplayTimestamp(readings)

longReplayJob(every 60 min):
    readings = selectReadings(last 60 minutes)
    sendBroadcastBatch(readings)
    updateReplayTimestamp(readings)

onAppRestart():
    readings = selectReadings(last 60 minutes)
    sendBroadcastBatch(readings)
```

## Final Recommendation

Treat xDrip broadcast mode as a **one-way, non-acknowledged, eventually consistent ingestion channel**.

### Best practical strategy

- persist every reading locally
- send once immediately
- replay recent windows periodically
- recover with a restart backfill window
- monitor sender health continuously

This approach is simple, operationally robust, and suitable for long-running unattended use.
